Audit

A read-only review of riptapparel.com built from the live WooCommerce system report. The site is running on modern infrastructure, but the database and media library have grown past healthy limits, and several active plugins are duplicating work. This document lists what we found, what to fix first, and a recommended order of operations for your team to follow.

Overview

The numbers behind the slowdown

Five data points from the live system report explain most of the speed and storage complaints. None of them require guessing. They are pulled directly from your WordPress install.

Database size

7.2 GB
Healthy WooCommerce stores this size sit closer to 1 to 2 GB.

Media attachments

548,954
Around 100 image files per product. Media library is the biggest storage cost.

Product variations

602,281
Expected for apparel (size x color x style), but adds DB load.

Active plugins

35
Several overlap in function. A tighter stack will be faster and easier to maintain.
Headline

The site is slow and the host is running out of space for two main reasons. First, the media library has grown to more than half a million files. Second, a handful of plugins (most notably the security plugin and the user database) have stored hundreds of megabytes of data they never clean up. Both are fixable without redesigning anything.

What is working
  • Server stack is modern: PHP 8.3, MariaDB 11.8, LiteSpeed web server, LiteSpeed Cache plugin installed
  • WooCommerce 10.7 with High Performance Order Storage already enabled, which is the right modern configuration
  • Payment stack is solid: WooPayments, Stripe, PayPal, Amazon Pay, Apple Pay and Google Pay all active
  • Hosting is on Hostinger with 15 GB memory and 15 GB upload limits available
What needs attention
  • Media library bloat driving the storage overload your manager has been fighting
  • Database bloat inside specific plugin tables (Wordfence, WooCommerce user meta, Action Scheduler history, FluentSMTP email logs)
  • Duplicated plugins doing the same job (two email platforms, three Elementor widget add-ons)
  • Elementor Pro licence is inactive, which means no security updates and silent break risk
  • No child theme, so any theme update will overwrite custom code including order email templates
  • Cart page has a conflict between a shortcode and a block, flagged by WooCommerce itself

Zero-risk fixes

Quick wins we can action this week

These changes do not touch customer data, the checkout flow, or any visual element. They can be done directly in WordPress admin and reversed in one click if needed. Combined impact: faster dashboard, faster time to first byte, and a meaningful drop in hosting disk usage.

01 Delete inactive plugins

What: Remove the 4 inactive plugins (Customer Reviews for WooCommerce, Rich Showcase, RIPT Variant 410 Handler, Templately).

Impact: Cleaner admin, fewer cron jobs, removes abandoned DB rows.

Time: 5 minutes.

02 Clear Wordfence file cache

What: Wordfence > Tools > clear stored file list (wp_wfknownfilelist = 510 MB).

Impact: Reclaims half a gigabyte of database storage with zero functional change.

Time: 2 minutes.

03 Consolidate email platform

What: Decide between Klaviyo and Omnisend. Deactivate the loser and export its list to the keeper.

Impact: Half the tracking JS per page, one subscription fee instead of two, no more duplicate contacts.

Time: 30 minutes once the choice is made.

04 Reduce FluentSMTP log retention

What: FluentSMTP > Settings > change log retention from unlimited to 7 days.

Impact: Frees 100 MB+ and keeps the table small going forward.

Time: 2 minutes.

05 Empty trash and clean placeholders

What: Empty the 18 trashed products and set WooCommerce to auto-delete failed/placeholder orders after 30 days.

Impact: Smaller DB, cleaner admin.

Time: 10 minutes.

06 Renew or remove Elementor Pro

What: Reactivate the Elementor Pro licence or, if not renewing, plan to replace Pro widgets with free alternatives.

Impact: Restores security updates and support, removes the licence warning in admin.

Time: 5 minutes to renew. Longer if replacing.

07 Turn off WP debug mode

What: Flip WP_DEBUG to false in wp-config.php.

Impact: Stops writing debug.log on every request. Small speed win, cleaner disk.

Time: 2 minutes.

08 Trim scheduled action history

What: WooCommerce > Status > Scheduled Actions > clear the 149,389 completed entries older than 30 days.

Impact: Smaller Action Scheduler tables, faster admin.

Time: 5 minutes.

Expected result

Doing all eight of the above should reduce the database by roughly 800 MB to 1 GB, remove duplicate tracking scripts from every page, and bring admin speed back to normal. None of these changes affect product pages, the checkout, customer accounts, or existing orders.

Safe deployment

How to apply these fixes without breaking the live site

riptapparel.com takes real orders every day. A change made directly on the live site can break checkout, lose revenue, or corrupt data. The pattern below is the standard way to avoid that. Every fix in this audit beyond the Quick Wins should follow it.

Step 01
Clone the site to staging
Create a private copy of riptapparel.com using Hostinger's built-in 1-click staging (available on Business plan and above). If staging is not included, clone the site to a separate subdomain using a migration plugin like Duplicator or All-in-One WP Migration. Staging has the same data, plugins, and theme as live, but no customers see it.
Step 02
Back up before every change
Take a full file and database backup before touching anything, on staging or live. Store it off the server (UpdraftPlus, BlogVault, or host-native backup to an external location). If a change fails, a one-click restore returns the site to its previous state in under five minutes.
Step 03
Apply the change on staging and test
Make the change on staging only. Walk through the testing checklist below. If any test fails, investigate on staging and retry. The live site is never touched until staging passes every test.
Step 04
Push to live and monitor for 24 hours
Once staging is verified, push the change to live using the host's push-to-live tool or a migration plugin. Monitor the WooCommerce error log, GA4 revenue, and customer support channels for 24 hours before making the next change.
What to test after every change

Homepage loads correctly. Category and product pages load. Add to cart works. Cart calculates totals, coupons, and shipping correctly. Checkout completes with a test card in WooPayments test mode. Order confirmation email arrives. Customer login and account pages load. No new entries in the WooCommerce error log.

Next step

Recommended first moves

This audit is a recommendation document, not a scope of work. The items below are the safest place for the RIPT team to start. The Quick Wins can be actioned in under an hour on the live site. Everything beyond that should move through a staging environment first.

01 Share this report internally

Send this audit to the developer or agency that maintains riptapparel.com. Align on which items to action first. The Quick Wins section is the lowest-risk starting point and should free roughly 800 MB to 1 GB of database storage immediately.

02 Confirm staging availability

Check that the current Hostinger plan includes staging. Most changes beyond Quick Wins should happen on a staging clone first, not on the live site. If staging is not included in the current plan, a short upgrade or migration is advisable before deep-clean work begins.

Headline recommendation

Start with the 8 Quick Wins this week. They are zero-risk and do not require staging. Every other item in this report should move through a staging environment with a full backup taken before each change.
